# LifeScript ハッカソン戦略

## 現状の強みと弱みの整理

### 実装済みの強み
- DSL → LLM コンパイル → RestrictedPython サンドボックス実行の一気通貫パイプライン
- traits / メモリ / マシン観察の3層パーソナライズ基盤
- 即時実行（Skills）と自動化（Automation）の静と動
- `widget_show()` による動的ホーム画面生成
- コンシェルジュチャットでの即時アクション実行
- カレンダー・通知・提案の統合UI

### 現状の課題
- デモで「すごい」と思わせる決定的な瞬間が設計されていない
- 社会的価値（Value）の言語化が弱い
- 技術の深さが審査員に伝わる構成になっていない

---

## 審査基準ごとの戦略

### 1. Value — 開発テーマに沿って課題を解決し、価値を生んでいるか

**課題設定**: 「AIに生活を任せる」のではなく「AIが自分の延長として動く」ことの価値

**打ち出すべき価値**:

- **透明性という社会価値**: AIの判断根拠がユーザー自身の言葉（traits）に遡れる。ブラックボックスAIの対極
- **エージェンシーの保証**: 「自分で書いたから、自分で直せる」— AIが間違えたとき、traits やメモリを編集すれば直る。受動的なAI利用ではなく、ユーザーが主権を持つ
- **漸進的な信頼構築**: 書けば書くほどマシンが賢くなる正のフィードバックループ。最初は浅い記述で基本動作、深い記述でより文脈的な提案

**具体的に伝えるストーリー**:
> 「バイトが週4入っていることに気づいて休息を提案した。その根拠は、ユーザー自身が書いた『バイトの許容は週3まで』というtraitだった。AIが勝手に判断したのではなく、ユーザーの言葉を根拠にしている。だから納得できるし、違ったら自分で直せる」

**アクション**:
- [ ] デモ用のtraitsとカレンダーデータを事前に仕込む（説得力のあるシナリオ）
- [ ] 提案の「理由」表示を目立たせ、trait引用をハイライトする

---

### 2. Technology — 技術選定が適切で、技術が有効に活用されているか

**見せるべき技術的深さ**:

| レイヤー | 技術 | 審査員に伝わるポイント |
|---|---|---|
| DSLコンパイル | LiteLLM + Gemini 2.5 Flash | LLMをランタイムではなくコンパイル時に限定 → コスト効率の設計判断 |
| セキュリティ | AST静的解析 + RestrictedPython | ユーザー入力をLLMが変換したコードを安全に実行する2層防御 |
| スケジューリング | APScheduler | cron/interval/onceの3パターン。エラー時の自動再コンパイル |
| パーソナライズ | traits + memory + memory_auto | 3層の文脈蓄積。LLMプロンプトに全て統合 |
| UI | Flet (Python) | Pythonのみでフルスタック。バックエンドとUIの統合が速い |
| DB | Supabase + SQLite フォールバック | クラウド本番 / ローカル開発の自動切替 |

**技術的に光る設計判断**:
- コンパイル時のみLLM → 実行時コストゼロ（他のAIアプリとの差別化）
- `widget_show()` でユーザーがUI自体を拡張できる（開発者が想定していない使い方を許容）
- マシンの「観察」機能: LLMがカレンダーパターンを読み取り、自動でメモリに蓄積

**アクション**:
- [ ] アーキテクチャ図を1枚用意する（コンパイルフロー + 実行フロー + パーソナライズフロー）
- [ ] 「LLMコストの設計」を数字で示す（1コンパイル = 1 API call、実行時 = 0 call）

### 3. UX — ユーザーが価値を自然に体験できる設計になっているか

**現状のUXの強み**:
- ホーム画面でカレンダー + 提案 + 通知が一覧
- 提案カードの承認/却下がワンクリック
- メモリが2箇所（ホーム + Machine）からアクセス可能
- チャットで「予定入れて」と言えば即実行

**改善すべき点**:

1. **初回体験の設計が不在**
   - 初めて開いたユーザーが何をすべきかわからない
   - オンボーディングでtraitsを自然に書かせる導線が必要

2. **「書けば賢くなる」の実感が薄い**
   - traitsを書いた直後に提案が変わる瞬間を見せる仕組みがない
   - 書く前と後の差分が視覚化されていない

3. **デモフローの最適化**
   - 審査員は5分で理解する必要がある
   - 「traits書く → 提案出る → 承認する → カレンダーに反映」の一連を2分で見せられるか

**アクション**:
- [ ] デモ用のオンボーディング的初回画面を簡易実装（traitsテンプレート入力）
- [ ] traits追加直後に即座に分析を走らせ、提案が出る即時フィードバックを実装
- [ ] Machineチャットで「自己紹介して」と言うとメモリの内容を要約して返す機能

### 4. Execution — アイデアが動くプロトタイプとして具現化されているか

**現状の完成度**: 高い。コア機能は全て動作する

**デモで壊れやすいポイントと対策**:

| リスク | 対策 |
|---|---|
| LLM API の遅延/エラー | SQLiteフォールバック済み。デモ用にコンパイルキャッシュを事前ウォームアップ |
| 提案生成に時間がかかる | デモ用データを事前に`machine_logs`に仕込む |
| UIが更新されない | ログポーリング間隔は1秒。手動リフレッシュボタンも配置済み |

**デモの流れ（推奨）**:
1. ホーム画面を見せる — 「これが今のマシンの状態です」
2. メモリを開く — 「マシンはこれだけのことを把握しています」
3. 提案カードを見せる — 「この提案の理由はユーザーのtraitsに基づいています」
4. IDEでtraitsを追加 — 「新しい文脈を教えます」
5. Runで即時実行 — 「SkillsとしてIDEから直接実行もできます」
6. チャットで予定追加 — 「自然言語でも操作できます」

**アクション**:
- [ ] デモシナリオを固定し、データを事前準備するスクリプトを作る
- [ ] デモ用のtraitsとカレンダーイベントのセットを用意
- [ ] 各画面遷移をスムーズにするためのUIアニメーション確認

### 5. Pitch — プロトタイプの魅力や価値が説得力を持って伝えられているか

**ピッチの核心メッセージ**:

> **「毎回話しかけなくていいChatGPT」**
>
> ChatGPTは毎回文脈を伝え直す。LifeScriptは一度書けば、マシンがずっと覚えている。
> しかもその記憶はブラックボックスではない——ユーザー自身の言葉で書かれたtraitsとメモリが根拠になっている。

**ピッチ構成案（5分）**:

| 時間 | 内容 |
|---|---|
| 0:00-0:30 | 問題提起: 「AIは便利だが、毎回文脈を伝えるのは面倒。自動化ツールは融通が利かない」 |
| 0:30-1:00 | コンセプト: 「文脈を一度書けば、マシンが生活を把握して動く」 |
| 1:00-3:00 | デモ: traits → 提案 → 承認 → チャット → Skills実行 |
| 3:00-4:00 | 技術: コンパイル時LLM・サンドボックス・3層パーソナライズ |
| 4:00-4:30 | 差別化: vs Notion (動的実行) / vs ChatGPT (永続文脈) / vs IFTTT (LLM理解) |
| 4:30-5:00 | ビジョン: 関数追加 = ロードマップ。天気・位置情報・習慣で世界が広がる |

**差別化の一言**:

| 比較対象 | LifeScriptの違い |
|---|---|
| Notion | ユーザーのコードが実際に動く。静的な情報管理ではなく動的な生活実行 |
| ChatGPT | 文脈が永続する。毎回説明し直さなくていい |
| IFTTT/Zapier | LLMが自然言語を理解してコンパイルする。トリガー設定の知識不要 |
| リマインドアプリ | ユーザーが設定していない行動まで提案する。根拠はユーザーの言葉 |

**アクション**:
- [ ] 上記構成でピッチ資料を作成
- [ ] デモ動画をバックアップとして録画
- [ ] 「3層パーソナライズ」の図を1枚作る（traits → メモリ → 観察 → 提案）

## 優先度順アクションリスト

### 最優先（デモの説得力に直結）
1. デモ用データ準備スクリプトの作成
2. traits追加 → 即時分析 → 提案生成の即時フィードバック実装
3. ピッチ資料の骨格作成

### 高優先（完成度の向上）
4. 簡易オンボーディング画面（traitsテンプレート入力）
5. アーキテクチャ図の作成
6. デモシナリオのリハーサル

### あれば良い（磨き）
7. UIアニメーションの微調整
8. エラーハンドリングの最終確認
9. Machineチャットの応答品質チューニング（プロンプト調整）
